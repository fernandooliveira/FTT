    CSS

    Overview

    • /public/stylesheets/lib/
    • /public/stylesheets/

    There are two parts to our CSS: the CSS library, and CSS that is specific to the application.

    The CSS library is written to a method loosely termed 'Object Oriented CSS'. What this means is that 'base' classes are defined to establish basic styles, and these base classes are then extended with 'sub' classes that create progressively more complex styles.

    The idea is to produce useful, reusable basic styles with plain language class names that can be intuitively added to html to bring it alive, while keeping the codebase small (DRY), scaleable, semantically understandable (KISS), and easily maintainable.

    Application-specific CSS define styles specific to parts of the application. When rules are frequently repeated, consider whether they should be moved to the library as reusable classes.

    Tests

    • /test/

    Effort needs to be made to keep tests up-to-date. They act both as a reference when looking for html constructs for achieving specific styles, and as tests when styles break. When you write an extension to an existing base class, add examples of that extension to the base class's test page (include images in the images/ sub-directory when necessary).

    New base classes should have a new directory created for them, named css-[name]/, where [name] is the name of the base class or html tag the test is written for.

    Code Style

    • Class (and id) names are in American English.
    • Never use an #id in the library. The library is for reusable code.
    • Class names are short and descriptive, using a single word where possible.
    • Multiple word class names are hyphenated: .my-class-name
    • Code is indented with 4 spaces.
    • Normally, CSS definitions are written one-per-line*: .class {
          font-family: 'Comic Sans MS', sans-serif;
          line-height: 1.5;
      }       but where it makes sense, groups of properties that are related can be placed on a single line: .class {
          position: absolute;
          top: 20px; right: 0; bottom: 20px; left: 0;
      }       and repetitive CSS is written on single lines, with brackets and properties spaced so they align vertically: .class-one   { background-color: black; color: #333333; }
      .class-two   { background-color: grey;  color: #222222; }
      .class-three { background-color: red;   color: #000000; }
    • Dimensions are written in the order x, y: .class {
          width: 200px; height: auto;
      }       while positions are written in clockwise order starting from the top: .class {
          top: 20px; right: 0; bottom: 20px; left: 0;
      }
    • Dimensions that are visually dependent on text size should be declared in em. The base font size is 12px, so 1em is equivalent to 12px (unless another font-size has been inherited from a parent): .class p {
          margin-bottom: 1.2em;
      }
    • IE7 does not support it, so the :last-child pseudo selector should be 'backed up' with the class .last: .class:last-child,
      .class.last {
          margin-bottom: 0;
      }
    • Comment unusual code. CSS, particularly browser-specific CSS, does not always explain itself very well.

    IE6 is not supported so you can freely use child > and attribute [attr=value] selectors. For support of other selectors check with a browser support matrix such as the one at QuirksMode:

    www.quirksmode.org/css/contents.html

    Best practises

    • Avoid htmltag.class or htmltag#id constructs. In general, defining html tag names as well as classes or ids is unnecessary, and only slows down the browser.
    • Avoid * selectors at the ends of style rules. Selecting everything is expensive, and can slow down the browser. Selection engines work right-to-left.
    * Note: getting caught using Comic Sans MS is grounds for instant dismissal.
